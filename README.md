# Provenance Guard

Provenance Guard is a backend service for classifying submitted text as likely human-written or AI-generated. The goal is not only to make a classification, but to expose why the decision was made, how confident it is, and to allow creators to appeal uncertain outcomes.

## Architecture overview

A submission flows through three stages:

1. `POST /submit` receives text plus `creator_id`.
2. The service runs two independent detection signals:
   - a Groq-style generative-model likelihood signal,
   - a stylometric heuristic signal.
3. The scores are fused into a posterior estimate with an uncertainty interval, mapped to a transparency label, stored in the audit log, and returned to the caller.

This path was chosen so the system combines semantic and surface-level evidence, then reports both the score and the interval width instead of hiding uncertainty.

## Detection signals

### Signal 1: Groq-like semantic probability
- Measures: whether the text matches AI-style patterns in meaning, phrasing, and tone.
- Why chosen: a modern LLM-based signal can detect the holistic style of generated text, which is hard to capture with simple heuristics alone.
- What it misses: adversarially written text that mimics human tone, or short snippets where the model has little context.

### Signal 2: Stylometric heuristics
- Measures: surface statistics such as type-token ratio, sentence-length variance, and punctuation density.
- Why chosen: these structural features are complementary to semantic signals and can detect repetitive or mechanically consistent output.
- What it misses: intentionally edited or human text that looks formulaic, and naturally sparse creative writing with low token counts.

## Confidence scoring

The two signal scores are converted to pseudo-counts and fused as a Bayesian posterior. Each signal contributes evidence: the Groq-style score is weighted more heavily, and the stylometric score provides a second perspective. The result is:

- `combined.mean` = posterior AI-likelihood
- `lower90`, `upper90` = 90% credible interval bounds
- `width` = upper90 - lower90

This makes confidence meaningful because decisions require both a strong mean and a narrow interval.

### Example submissions

- High-confidence AI example:
  - `model_score = 0.9907`
  - `heuristic_score = 0.5286`
  - combined posterior = `mean 0.8573`, `lower90 0.7696`, `upper90 0.9450`, `width 0.1755`
  - label: `High-confidence AI`

- Lower-confidence example:
  - `model_score = 0.3548`
  - `heuristic_score = 0.5286`
  - combined posterior = `mean 0.4031`, `lower90 0.2800`, `upper90 0.5261`, `width 0.2461`
  - label: `Uncertain`

These examples show that the scoring is not constant: the same heuristic signal can produce different transparency outcomes depending on the model score and interval width.

## Transparency labels

All three variant texts are documented exactly:

- `High-confidence AI — This content is very likely generated by AI. The system's confidence is {confidence:P0.0} (lower-upper).`
- `High-confidence human — This content is very likely written by a human. The system's confidence that this is AI-generated is {confidence:P0.0} (lower-upper).`
- `Uncertain — The system is uncertain whether this content was written by a human or generated by AI. The system's AI-likelihood is {confidence:P0.0} (lower-upper).`

The system currently returns a concise label, and these exact phrases are the intended display variants for the transparency layer.

## Rate limiting

The service applies rate limits to protect the endpoint from abuse and preserve audit capacity:

- `10 requests per minute` per `creator_id`
- `100 requests per day` per `creator_id`

These values balance responsiveness for legitimate users with protection against repeated bulk submission attacks. The minute-level burst cap prevents quick retries and the daily cap limits long-term scraping.

Running `test_rate_limit.py` demonstrates what happens when the limit is exceeded. The first 10 requests succeed with `200 OK`; the 11th returns `429 TOO MANY REQUESTS`:


## Audit log and appeals

Every submission is written to `audit.log` as a JSON line containing:

- `content_id`
- `creator_id`
- `timestamp`
- raw signal outputs
- combined posterior and interval
- assigned label
- current status

The system also supports `POST /appeal`, which records a creator appeal, marks the content as `under_review`, and appends an audit event.

## Known limitations

- Short or repetitive text can be misleading. If a human writes a short poem or tweet with simple vocabulary, the stylometric signal may appear AI-like because it has low type-token ratio and low sentence variance.
- The current implementation uses a mock Groq score and JSON file persistence, so the real system would need a calibrated LLM service and stronger storage for production.
- Inputs with code, quoted passages, or highly domain-specific jargon are not specially handled, which can shift both signals away from valid human style.

## Spec reflection

### What the spec helped with
The planning doc made the architecture explicit: separate submission, signal fusion, audit logging, labels, and appeals. That guided the implementation to keep auditing and uncertainty reporting central rather than optional.

### Where implementation diverged
The spec described a full SQLite-backed review queue and admin PATCH workflow. In this version, I simplified persistence to a JSON-backed content store and implemented `POST /appeal` only, because it kept the core production flow working while still supporting review state.

## AI usage

Two specific AI-driven development steps:

1. I directed the AI tool to generate a Flask skeleton with `POST /submit`, structured JSON audit logging, and signal fusion helpers. I then revised the generated code to add persistent content storage, combined score thresholding, and appeal-state handling.
2. I asked the AI tool to explain and document the exact transparency label variants. The initial label wording was too generic, so I overrode it with explicit, user-facing phrases and added interval-aware human/AI wording.

## Portfolio walkthrough

1. A client sends text to `POST /submit` with `creator_id`.
2. The backend computes two independent signals and fuses them into a posterior probability plus a 90% interval.
3. If the posterior is extreme enough and the interval is narrow enough, the content is labeled `High-confidence AI` or `High-confidence human`; otherwise it is labeled `Uncertain`.
4. The decision is written to `audit.log` and saved in `content_store.json`.
5. If the creator disagrees, they call `POST /appeal`, which updates status to `under_review` and records the appeal event.

This README focuses on why the design was chosen, not just what was built, and it includes real score examples to show the system produces meaningful variation in confidence.
