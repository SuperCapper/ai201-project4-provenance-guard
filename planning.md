# Provenance Guard — Planning

This document is the implementation-ready spec and prompt for AI-assisted development. It answers the five required questions with concrete, testable choices, includes the architecture diagram, and specifies what to ask AI tools for each milestone.

## 1) Detection signals

- **Signal 1 — LLM-based classification (Groq)**
  - What it measures: semantic and stylistic likelihood the text was produced by a modern generative model. The service will send a compact prompt to Groq (llama-3.3 style) asking for a probabilistic assessment and short rationale.
  - Output shape: JSON {"model_score": float in [0,1], "explanation": string, "prompt_hash": string}
  - Interpretation: `model_score` is the model's estimated probability that the text is AI-generated.
  - Blind spots: susceptible to adversarial prompt engineering and examples that deliberately mimic model training data; confidence calibration is needed.

- **Signal 2 — Stylometric heuristics (pure Python)**
  - What it measures: surface statistical properties of writing (sentence length variance, type-token ratio, punctuation density, average clause/sentence complexity, stopword distribution). These capture structural regularities common in model outputs.
  - Output shape: JSON {"heuristic_score": float in [0,1], "features": {"tok_count":int, "type_token_ratio":float, "sent_len_var":float, ...}}
  - Interpretation: `heuristic_score` maps higher values to higher likelihood of AI generation based on empirical thresholds set from a small validation set.
  - Blind spots: short texts (<100 tokens) have noisy statistics; adversaries can mimic surface statistics; does not capture semantic coherence or topical oddities.

- **Why these two:** one is semantic/holistic (Signal 1), the other structural/statistical (Signal 2). They are intentionally different to reduce correlated failure modes.

## 2) Uncertainty representation & scoring

We represent uncertainty explicitly as a combined posterior mean and a 90% credible interval (lower, upper). Implementation strategy:

- Normalize both `model_score` and `heuristic_score` to be calibrated probabilities in [0,1] using held-out calibration data and Platt scaling (logistic regression) fit on validation labels.
- Combine scores using a simple Bayesian fusion model: treat each calibrated score as an independent noisy observation of a Bernoulli parameter p with Beta priors. Concretely:
  - For each signal i, convert calibrated score s_i to pseudo-counts: alpha_i = s_i * k_i, beta_i = (1 - s_i) * k_i, where k_i is an evidence weight (initial default: k_model=30, k_heuristic=10 — model more weight but configurable).
  - Posterior Beta parameters: alpha = alpha_model + alpha_heuristic + alpha_prior, beta = beta_model + beta_heuristic + beta_prior (alpha_prior=1, beta_prior=1).
  - Combined score = posterior mean = alpha / (alpha + beta).
  - Uncertainty interval (90% credible interval) = Beta.ppf(0.05, alpha, beta), Beta.ppf(0.95, alpha, beta).

Mapping to labels (concrete thresholds):

- **High-confidence AI** — combined_mean >= 0.85 AND (upper - lower) <= 0.20
- **High-confidence Human** — combined_mean <= 0.15 AND (upper - lower) <= 0.20
- **Uncertain** — otherwise (covers middle probabilities and cases with high interval width)

Notes:
- The uncertainty width check prevents narrow thresholds from producing high-confidence labels on noisy inputs. The numeric thresholds (0.85/0.15 and 0.20 width) are concrete starting points; we'll validate and adjust using a small calibration/dev set.
- For very short texts (<100 tokens) we reduce k_i (evidence weight) to reflect lower signal reliability and therefore automatically widen uncertainty intervals.

Example: model_score=0.9 (k=30) -> alpha_model=27 beta_model=3; heuristic_score=0.6 (k=10)-> alpha_h=6 beta_h=4; posterior mean ~ (27+6+1)/(27+3+6+4+2)=34/42=0.81 -> uncertain if interval wide; with tighter evidence could be high-confidence AI.

## 3) Transparency label design (exact text)

Write the verbatim label text now (these exact strings will appear in the README and be returned by `GET /label-variants`):

1) "High-confidence AI — This content is very likely generated by AI. The system's confidence is {confidence:P0.0} (lower-upper)."

2) "High-confidence human — This content is very likely written by a human. The system's confidence that this is AI-generated is {confidence:P0.0} (lower-upper)."

3) "Uncertain — The system is uncertain whether this content was written by a human or generated by AI. The system's AI-likelihood is {confidence:P0.0} (lower-upper)."

Notes:
- `{confidence:P0.0}` is a formatted probability (e.g., 0.62) and `lower-upper` are the 90% credible interval bounds; the UI can replace these placeholders with numbers.
- The phrasing emphasizes plain-language explanation and includes the numeric score and interval for transparency.

## 4) Appeals workflow

- **Who can submit:** the content creator (authenticated by platform) or a platform admin. The `POST /appeal` endpoint accepts an appeal; the minimal required fields: `{ request_id, creator_id, reason_text, contact_email (optional) }`.
- **What the system does on receipt:**
  - Validate the request_id exists and the appeal window (e.g., 30 days) is open.
  - Create an audit entry: `{appeal_id, request_id, creator_id, reason_text, created_at, initial_status: "under_review"}`.
  - Update content status to `under_review` in the persistent store.
  - Notify human moderation queue (simple database queue entry) and return `202 Accepted` with `appeal_id`.
- **Status changes & logging:** statuses: `under_review`, `needs_info`, `rejected`, `upheld` (reviewer agrees content is AI), `overturned` (reviewer marks human). Every status change appends a log entry with reviewer id, timestamp, notes, and updated fields.
- **What a human reviewer sees:** original text (or text hash with ability to view content per privacy rules), original label, combined score and interval, signal-level details (model_score, heuristic features), appeal text, and previous audit entries. Reviewer has UI to set final decision and notes.
- **No automated reclassification:** the system will not automatically change the label without an explicit human decision; appeal outcomes are recorded in the audit log and optionally surfaced to the platform.

## 5) Anticipated edge cases (specific scenarios)

1) **Very short creative pieces** (e.g., a 30-word micro-poem): stylometric heuristics are unreliable due to insufficient tokens; both k_model and k_heuristic are reduced and the result will often be `Uncertain`.

2) **Poems with heavy repetition and simple vocabulary**: heuristics (low type-token ratio, low sentence variance) might push `heuristic_score` toward AI; however, many human-written poems intentionally use repetition. These should often remain `Uncertain` unless the model signal is strongly AI-like; this will be flagged in the README as a failure mode.

3) **Prompt-engineered text that mimics human style**: an adversary can craft text to lower model_score while preserving surface statistics; combined fusion helps but this remains a risk — logged as "adversarial mimicry" in the audit.

4) **User-provided content that includes copyrighted snippets or code blocks**: preprocessor must detect long verbatim quotes or code blocks and either exclude them or treat them specially (short-circuit to `Uncertain` and log the reason).

## ## Architecture

Submission flow (2–3 sentence narrative): Client `POST /submit` -> rate limiter -> preprocessor -> parallel signals (Groq LLM and stylometric heuristics) -> combiner/calibration -> labeler -> audit log -> response. Audit entries include full signal outputs and the combined posterior used to pick the label.

Appeal flow (2 sentences): Creator `POST /appeal` with `request_id` -> appeal handler saves appeal and sets content status `under_review`, adds audit entry, and enqueues human review. Reviewer updates status via `PATCH /appeal/{id}`; change is logged and stored.

## AI Tool Plan

This section defines exactly what to give an AI code-generation tool (the spec sections above + the diagram) for each milestone, and how to verify outputs.

- **M3 (submission endpoint + first signal)**
  - Provide: `Detection signals` section (Signal 1 only) + `Architecture` diagram + `POST /submit` contract.
  - Ask the AI tool to generate: a minimal Flask app skeleton with `POST /submit` that accepts JSON `{text, creator_id, metadata}` and returns a placeholder response. Implement `signal1_groq(text)` function that calls the Groq client (mockable) and returns `{model_score, explanation, prompt_hash}`. Add structured audit logging calls (write JSON lines to `audit.log`). Add simple rate limiting decorator using `Flask-Limiter` configured for per-creator limits.
  - Verification: run the Flask app locally and call `POST /submit` with five test inputs (human sample, clearly AI sample, short text, poem, long blog post). Confirm the response contains `model_score` and `request_id` and that `audit.log` has entries for each request.

- **M4 (second signal + confidence scoring)**
  - Provide: `Detection signals` (both signals) + `Uncertainty representation` + `Architecture` diagram.
  - Ask the AI tool to implement `signal2_stylometric(text)` that returns the feature vector and a calibrated `heuristic_score`. Implement calibration helpers and the Bayesian fusion combiner that returns `{combined_mean, lower90, upper90}`.
  - Add logic in `POST /submit` to run both signals in parallel (threadpool or simple sequential calls), compute combined posterior, select label per thresholds in this doc, and return detailed signal outputs.
  - Verification: use a labeled test set of 10 examples (we will create a small in-repo `tests/samples.json`) and show that clearly AI texts map to high combined_mean, clearly human map low, and edge cases produce `Uncertain` labels. Confirm credible intervals behave as signal strength varies.

- **M5 (production layer: labels, appeals, rate limits, audit log)**
  - Provide: `Transparency label design` (verbatim strings), `Appeals workflow`, `Architecture` diagram, and `Anticipated edge cases`.
  - Ask the AI tool to implement endpoints:
    - `POST /appeal` (creates appeal, returns `appeal_id`)
    - `GET /appeal/{id}`
    - `PATCH /appeal/{id}` (admin only — implement simple API key auth configurable via `.env`)
    - `GET /log` (paginated read of audit entries)
    - `GET /label-variants`
  - Add persistent storage (SQLite) for requests, appeals, and audit records (schema provided in the ask). Add unit tests that simulate an appeal flow: submit content, get label, submit appeal, review appeal, assert DB and log entries are correct and status updated to `under_review` then to final state.
  - Verification: run the full stack locally; perform an end-to-end test (submit -> receive label -> submit appeal -> reviewer PATCH -> status updated). Confirm `audit.log` shows at least three entries for the cycle: initial decision, appeal creation, reviewer decision.

## Implementation notes and safety

- Store secrets in `.env`. Keep `.env` in `.gitignore`. Use `python-dotenv` to load keys.
- Rate limits: default per-creator 60 requests / hour, burst 10/min. Document in README.
- Audit log schema: `{ request_id, timestamp, action, actor, raw_text_hash, signals: {...}, combined: {...}, label, metadata }`.
